# Redveil

`Redveil`은 서울 소형 상가 매입 리스크 판별 서비스입니다.  
핵심은 `공공 데이터 기반 리스크 모델 -> 문장형 리스크 메모 -> 비교 가능한 대체 후보 -> 공개 웹사이트` 흐름을 하나의 제품 경험으로 연결한 점입니다.

## 프로젝트 한눈에 보기

| 항목 | 내용 |
|------|------|
| 도메인 | 부동산 / 상권 분석 / 투자 의사결정 지원 |
| 진행 형태 | 개인 프로젝트 |
| 문제 정의 | 서울 소형 상가 투자자가 매입 전 공실 위험, 상권 과열 위험, 가격 부담 위험을 한 번에 점검할 수 있는가 |
| 핵심 방법 | 실거래가 수집, 상권 데이터 결합, 규칙 기반 리스크 스코어링, 리스크 유형화, 웹사이트 구현 |
| 주요 도구 | Python, Pandas, scikit-learn, Streamlit, HTML/CSS/JavaScript, GitHub Pages |
| 핵심 결과 | 서울 25개 구를 대상으로 매입 리스크를 점수화하고, 매물 검토 메모와 대체 후보 비교까지 가능한 웹서비스 형태로 구현 |
| 공개 링크 | [GitHub 저장소](https://github.com/dffxonnb-cyber/Seoul-Storefront-Redveil), [웹사이트](https://dffxonnb-cyber.github.io/Seoul-Storefront-Redveil/) |

## 이 프로젝트가 해결한 질문

- 어떤 구가 단순히 인기 있는 곳이 아니라 `실제로 매입을 보류해야 할 위험 구`인가
- 같은 총점이라도 `고가 선행형`, `거래 둔화형`, `과밀 경쟁형`처럼 위험 패턴이 다르게 설명될 수 있는가
- 대시보드 수준을 넘어서 `내가 지금 검토 중인 매물` 관점의 판단 도구로 만들 수 있는가

## 접근 방식

1. 국토교통부 상업업무용 실거래가 API로 최근 12개월 서울 거래 데이터를 수집했습니다.
2. 서울시 상권분석서비스와 소상공인시장진흥공단 데이터를 결합해 상권 과밀도와 수요 취약 신호를 만들었습니다.
3. 가격 부담, 거래 유동성, 가격 변동성, 상권 과밀을 축으로 규칙 기반 리스크 점수를 설계했습니다.
4. 구 단위 종합 점수만 주지 않고, 리스크 유형과 현장 검증 체크리스트를 함께 생성했습니다.
5. 결과를 `3분 진단`, `내 매물 검토`, `후보 비교`, `구별 리포트`로 이어지는 공개 웹사이트로 구현했습니다.

## 핵심 결과

- 서울 `25개 구`를 동일한 규칙으로 비교 가능한 매입 리스크 모델로 정리했습니다.
- 행정동 `428개`, 수요 취약 상권 `1,570개`를 참고 축으로 연결했습니다.
- 실거래 원천 `12,074건`을 바탕으로 최근 12개월 거래 리스크를 만들었습니다.
- 리스크 유형을 `고가 선행형`, `거래 둔화형`, `과밀 경쟁형`, `가격 변동형`, `복합 점검형`으로 분류했습니다.
- 저장 가능한 `매물 검토 메모`와 `대체 후보 비교` 기능까지 포함한 정적 웹사이트를 구성했습니다.

## 포트폴리오 관점에서 강한 이유

- `기획 -> 데이터 수집 -> 분석 모델 -> 서비스 구현 -> 공개 배포` 흐름이 닫혀 있습니다.
- 단순 시각화가 아니라, 실제 투자 판단을 바꾸는 문장형 경고와 행동 가이드를 제공합니다.
- 분석 결과를 웹서비스와 GitHub Pages 배포까지 연결해 `기획 + 개발 + 분석` 역량을 함께 보여줍니다.

## 이 저장소에서 보여주려는 역량

- 여러 공공 데이터를 결합해 리스크 모델을 설계하는 능력
- 점수만이 아니라 `설명 가능한 반대 근거`로 번역하는 해석력
- 분석 결과를 실제 서비스 흐름으로 바꾸는 제품 기획력
- 로컬 프로토타입을 공개 가능한 정적 웹사이트로 정리하는 구현력

## 폴더 구조

| 경로 | 설명 |
|------|------|
| `app/site/` | 공개 웹사이트 HTML, CSS, JS |
| `app/server.py` | 로컬 검증용 Python 서버 |
| `src/redveil/pipelines/` | 데이터 수집, 가공, 스코어링, 사이트 payload 생성 파이프라인 |
| `docs/` | PRD, 리스크 모델 정의, 사용자 여정, 검증 전략 문서 |
| `dashboard/` | 배포용 시각화 및 참고 자산 |
| `tests/` | 파이프라인 및 API 형태 테스트 |

## 실행 방법

1. `pip install -r requirements.txt`
2. 웹사이트 payload 생성

```bash
python src/redveil/pipelines/export_website_payload.py
```

3. 로컬 서버 실행

```bash
python app/server.py --host 127.0.0.1 --port 8030
```

4. 브라우저에서 아래 주소 접속

```text
http://127.0.0.1:8030
http://127.0.0.1:8030/review.html
http://127.0.0.1:8030/assessment.html
http://127.0.0.1:8030/compare.html
http://127.0.0.1:8030/districts.html
```

## 공개 저장소 기준 한계

- 원본 `data/` 폴더는 용량 문제로 GitHub 저장소에 포함하지 않았습니다.
- 현재 공개 저장소는 코드, 문서, 웹사이트, 시각화 중심으로 정리된 포트폴리오 버전입니다.
- 거래 데이터와 상권 수요 데이터의 시점이 완전히 일치하지 않아, 투자 의사결정의 1차 검토 도구로 보는 것이 적절합니다.
